
# ИИ-агент для многошагового поиска по технической документации



Исследовательский прототип для магистерской ВКР на тему:



**«Исследование методов построения ИИ-агента для многошагового поиска и анализа технической информации в открытой документации программных систем»**.



Проект предназначен для экспериментального сравнения методов поиска и генерации ответа по технической документации:



* BM25 RAG;

* Vector RAG;

* Hybrid RAG;

* Agentic RAG.



Основной фокус проекта — не пользовательский интерфейс, а воспроизводимый исследовательский стенд: загрузка корпуса документации, построение индексов, запуск разных retrieval-augmented методов, логирование результатов и последующий анализ качества ответов.



## Идея проекта



Большие языковые модели могут формировать связные ответы, но при работе с технической документацией важно, чтобы ответ был основан на проверяемых источниках. Обычный RAG-подход решает эту задачу частично: система ищет релевантные фрагменты документации и передаёт их в LLM как контекст.



Однако в технических задачах недостаточно просто найти похожие фрагменты. Система должна также определить:



* относится ли вопрос к доступному корпусу документации;

* достаточно ли найденного контекста для ответа;

* не требуется ли корректный отказ вместо генерации неподтверждённого ответа;

* какие источники использованы для формирования результата.



Для этого в проекте реализован Agentic RAG — ролевой многоэтапный контур, который выполняет дополнительные проверки до вызова языковой модели.



## Что реализовано



В проекте реализованы следующие компоненты:



* загрузка открытой документации FastAPI;

* очистка и фрагментация HTML-страниц;

* сохранение фрагментов документации с метаданными;

* построение BM25-индекса;

* построение векторного индекса;

* BM25 RAG;

* Vector RAG;

* Hybrid RAG;

* Agentic RAG;

* запуск экспериментов по тестовому набору вопросов;

* логирование retrieval-результатов, статусов, времени выполнения и agentic-шагов;

* анализ результатов эксперимента.

* Dependencies;

## Сравниваемые методы



### BM25 RAG



Лексический поиск по фрагментам документации.



BM25 хорошо работает с точными техническими терминами: именами классов, параметров, функций, команд и ошибок. Ограничение метода состоит в зависимости от формулировки запроса.



### Vector RAG



Семантический поиск по embedding-представлениям.



В проекте использовалась модель:



```text

sentence-transformers/all-MiniLM-L6-v2

```



Векторный поиск позволяет находить фрагменты, близкие по смыслу, даже если в вопросе и документации используются разные формулировки. Ограничение метода состоит в том, что семантическая близость не всегда означает достаточность фрагмента для точного технического ответа.



### Hybrid RAG



Гибридный поиск объединяет BM25 и векторный поиск.



В проекте использовалось объединение результатов через Reciprocal Rank Fusion. Цель метода — совместить точность лексического поиска и гибкость семантического поиска.



### Agentic RAG



Agentic RAG реализован как ролевой многоэтапный контур:



1. **Query Planner** — формирует дополнительные поисковые запросы.

2. **Retrieval Specialist** — выполняет многошаговый hybrid retrieval.

3. **Scope Checker** — проверяет, не выходит ли вопрос за пределы корпуса.

4. **Sufficiency Checker** — оценивает достаточность найденного контекста.

5. **Answer Generator** — вызывает LLM и формирует ответ.

6. **Verifier / Postprocessor** — проверяет и нормализует итоговый результат.



Не все роли реализованы как отдельные LLM-вызовы. Часть проверок выполняется алгоритмически, что повышает воспроизводимость и позволяет не вызывать языковую модель при заведомо нерелевантном вопросе.



## Эксперимент



Для пилотной оценки использовался тестовый набор из 6 вопросов:



* настройка CORS;

* объявление path parameter `item_id` как `int`;

* создание зависимости через `Depends`;

* сборка Docker image для FastAPI-приложения;

* назначение `OAuth2PasswordBearer` и `tokenUrl`;

* вопрос про Django ORM как пример недостаточной информации в корпусе.



Вопрос про Django ORM использовался для проверки корректного отказа: выбранный корпус документации FastAPI не содержит достаточного основания для ответа на него.



## Результаты пилотного эксперимента



В рамках пилотного эксперимента были получены следующие результаты:



* baseline-методы BM25 RAG, Vector RAG и Hybrid RAG дали 9 успешных ответов из 18;

* Agentic RAG дал 6 успешных ответов из 6;

* для вопросов с заданным ожидаемым источником все методы находили нужный источник: 20 из 20;

* основной причиной ошибок baseline-методов был не retrieval-этап, а генерация ответа и оценка достаточности контекста;

* для вопроса про Django ORM Agentic RAG сформировал корректный отказ без вызова LLM.



Эти результаты рассматриваются как пилотная экспериментальная проверка методики, а не как универсальное доказательство превосходства Agentic RAG во всех условиях.



## Практическая область применения



Наиболее перспективная область применения такого подхода — закрытые корпоративные контуры, где требуется работа со сложным инженерным доменом или специализированной информационной системой при высоких требованиях к информационной безопасности.



В таком сценарии корпус может формироваться из внутренней документации системы, регламентов, инструкций, справочников параметров, эксплуатационных материалов и базы типовых ошибок. ИИ-агент может использоваться как интеллектуальный консультант внутри программной системы, при этом ответ ограничивается заданным корпусом и сопровождается ссылками на источники.

```powershell

source .venv/Scripts/activate

```



## Локальная LLM



Для генерации ответов использовался локальный запуск модели через Ollama.



Основная модель эксперимента:



```text

qwen2.5:7b-instruct

```



Пример проверки Ollama:



```bash

ollama list

```



Перед запуском RAG-методов необходимо убедиться, что Ollama запущена и выбранная модель доступна локально.

## Ограничения



Проект является исследовательским прототипом и не является промышленной системой.



Основные ограничения:



* экспериментальный корпус ограничен документацией FastAPI;

* пилотный тестовый набор содержит 6 вопросов;

* ручная разметка качества выполнена экспертно;

* использовалась одна локальная LLM;

* результаты зависят от выбранной модели, корпуса и параметров запуска;

* пользовательский интерфейс не являлся основным объектом исследования.
